# データベース設計（SQLite）

最終更新: 2026-07-22

正本（source of truth）はnative-hostとTauriが共有DB層を通じて利用するSQLiteのみ（`docs/仕様書.md` 4.1節）。DDLは [`crates/engine-core/fixtures/schema.sql`](../crates/engine-core/fixtures/schema.sql)、サンプルデータは [`crates/engine-core/fixtures/seed.sql`](../crates/engine-core/fixtures/seed.sql) を参照。本ドキュメントは各テーブルの役割と関係を説明する。

接続時は必ず `PRAGMA foreign_keys = ON;` を実行すること。

`PRAGMA user_version` でスキーマ世代を管理する。v0からv1への起動時マイグレーションでは、`duplicate_members`をトランザクション内で再構築して`similarity`のCHECK制約を追加する。v1からv2では`courses.academic_year`と`courses.folder_name_override`を追加し、既存行に限って`term`先頭4桁から年度を補完する。以後の実行時処理は`term`から年度を推測しない。既存値を安全に移行できない場合はデータを変更・削除せず移行を中止し、DB接続エラーとして報告する。

## テーブル一覧

| テーブル                                     | 役割                                 |
|------------------------------------------|------------------------------------|
| `app_settings`                           | キー・バリューの単一設定（保存先ベースパス、アプリバージョン等）   |
| `extension_runtime_observations`          | 拡張機能のインストール・バージョン単位の初回／最終実応答 |
| `courses`                                | Moodleのコース一覧                       |
| `global_rule`                            | グローバルなフォルダ命名ルール（常に1行）              |
| `course_rule_overrides`                  | コース単位の例外ルール（回ごとに分けない等）             |
| `files`                                  | 保存済みファイルのメタデータ・ハッシュ・ルール適合状況        |
| `duplicate_groups` / `duplicate_members` | 重複・類似ファイルのグループ化                    |
| `assignments`                            | 課題・締切情報（ファイル/Moodleテキスト/ダッシュボード由来） |
| `notification_rules`                     | 締切通知のタイミング設定（複数登録可）                |
| `search_index_meta`                      | Tantivy全文索引の補助メタ情報                 |
| `sync_events`                            | Moodleからの課題・締切データ取得（同期）1回ごとの記録。データ取得通知の元データ |
| `assignment_changes`                     | 同期のたびに検出された課題の変更点（変更点表示の元データ）      |

## 関係の要点

- `extension_runtime_observations` はブラウザ名や導入済みフラグを持たず、拡張機能が生成したローカルな`installation_id`、`extension_version`、`protocol_version`の組み合わせごとに`first_seen_at`／`last_seen_at`を保持する。再インストールは新しい`installation_id`、更新は新しいバージョン行として区別できる
- 初期セットアップの拡張機能ステップは、確認画面の開始日時以降に記録された最新行を参照し、`protocol_version`が現在の契約バージョンと一致する場合だけ完了と算出する。過去の行が存在するだけでは完了にしない
- 応答日時は拡張機能の端末時刻ではなく、native-hostが受信時に付与したUTC時刻を正とする。同一インストール・バージョン・通信仕様からの再応答では`last_seen_at`だけを更新し、応答のたびに無制限に行を増やさない
- `files.course_id` は `courses.id` を参照。手動保存などコース不明のファイルは `NULL` を許容
- `courses.name`はMoodleから取得した生のコース名、`academic_year`は学期とは独立した年度の正本、`folder_name_override`はユーザーが変更した保存用フォルダ名だけを保持する。自動生成した保存名は生の名前・Moodle安定ID・全コースとの比較から決定的に導出できるため重複保存しない
- 実効コースフォルダ名は共有RustエンジンがNFKC正規化・Windows名検証・80 UTF-16コード単位への短縮・衝突回避を行って導出する。`folder_name_override`の更新は全コースを同じトランザクション内で再解決し、別コースが現在使用中の実効名を指定した場合、または大文字・小文字を区別しない実効名の重複が残る場合は拒否する。編集によって別コースの保存名を暗黙に変更しない
- `course_rule_overrides` は `courses` と1対1（`UNIQUE(course_id)`）。行が無いコースは `global_rule` のみ適用
- `files.rule_compliant = 0` のレコードが「ルール管理・整合性チェック画面」の警告一覧の元データ。理由は `violation_reason` に保持する。違反検出ロジックは `crates/engine-core` の `RuleEngine` が担当し、専用テーブルは持たずファイル行への注釈で表現する
- `assignments.due_at_status = 'needs_review'` が期限の異常検知（学期から大きく外れた締切）の対象
- `assignments.submission_mode` は提出状況の扱い方を表す：`moodle_auto`（Moodleから自動反映）／`manual`（ユーザーが手動で印）／`notify_only`（提出状況は追跡せず通知のみ）
- `duplicate_groups` は完全重複（`method='exact'`、`hash_blake3`一致）と類似（`method='similar'`、simhash近傍）の両方を表現できる。`duplicate_members.similarity` は `CHECK` 制約で0.0以上1.0以下に限定し、完全重複は1.0を保存する
- `notification_rules` は複数行登録でき、プリセットに加えてユーザーが締切時刻から365日前までの任意の相対時間を追加・削除し、個別に有効/無効化できる。`label`は`offset_minutes`から生成し、同じ相対時間を重複登録しない
- 保存先候補の絶対パスは`app_settings.base_folder_path`とルールから都度生成し、UIには保存ルート以下の相対パスだけを返す。警告・重複一覧でも、native-hostが`files.saved_path`から保存ルートを除いてファイル名込みの相対パスを導出する。保存ルート外の行は絶対パスを返さずエラーとして扱う。表示専用の相対パスは導出可能なためDBへ重複保存しない
- 警告一覧の`courseName`は`files.course_id`から`courses`を`LEFT JOIN`して導出し、DBへ重複保存しない。`course_id`とコース名は紐付け済みなら両方、未紐付けなら両方`NULL`相当としてAPIへ投影する
- `sync_events` は1回の同期（Moodleからの課題・締切データ取得）につき1行。`new/changed/removed_assignment_count` の集計値をデータ取得通知の文言（例:「3件の変更があります」）にそのまま使える
- `assignment_changes` は `sync_events` と `assignments` に紐づく1課題1フィールド1行の差分ログ。`field` に変更されたカラム名（`due_at` 等）、`old_value`/`new_value` に変更前後の値（表示用に文字列化したもの）を保持し、締切ハブの変更点一覧はこのテーブルをそのまま描画する

## サンプルデータの内容

`seed.sql` は Fuzzy.pdf のダッシュボードモックアップに登場する6科目（情報アーキテクチャ・データベース・離散数学・アプリ演習・認知科学概論・英語IIB）を使用し、以下のケースを含む。

- コース単位の例外ルール例：「アプリ演習」を回ごとに分けない設定
- ルール違反ファイルの例：規定外の場所に保存された資料
- 完全重複ファイルの例：同一資料を誤って2回保存したケース
- 締切の異常検知例：学期の範囲から外れた締切（`needs_review`）
- 期限切れの例：通知のみ対象の締切が過ぎたケース
- データ取得通知・変更点表示の例：直近の同期（`sync_events` id=2）で「認知科学概論 期末レポート」が要確認へ変化、「正規化レポート提出」の締切が1日延長されたケースを `assignment_changes` に記録

`bun:sqlite` のインメモリDBへDDLとseedを適用し、共有JSONのルール・警告・重複一覧との一致を自動テストする。Rust側でも外部キー制約と類似度の境界値を検証する。
