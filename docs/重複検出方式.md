# 重複・類似ファイル検出方式

最終更新: 2026-07-22

`DuplicateDetector`のヒューリスティック、SQLiteへの登録境界、今後の評価方法を定める。機能要件は[`仕様書.md`](仕様書.md)、DB上の表現は[`データベース設計.md`](データベース設計.md)を正とする。

## 1. 判定の優先順位

1. ファイルをストリーミングで読み、BLAKE3を計算する
2. `hash_blake3`が一致するファイルを`exact`として検出する
3. 完全一致しないファイルだけを、64 bit SimHashの近傍として`similar`判定する
4. 検出結果を提示・警告へ利用し、ファイルの移動・削除・統合は行わない

新規BLAKE3値は`b3:<64桁の小文字16進数>`形式とする。完全一致は内容の同一性を表すため、ファイル名・保存先・更新日時・MIME typeを判定に混ぜない。

## 2. SimHash特徴抽出

現段階では、形式固有のパーサーに依存しない次の方式を使う。

- 内容を8 byteの重なり窓として読む
- 窓のFNV-1a値に基づく内容依存サンプリングで約16分の1を特徴として採用する
- 採用した特徴を別の64 bitミキサーへ通し、各bitの多数決でSimHashを作る
- 小さいファイル等で特徴が選ばれなかった場合は、先頭の窓またはファイル全体の短い内容をフォールバック特徴にする
- BLAKE3とSimHashは同じストリーミング読み取りで計算し、ファイル全体をメモリへ載せない

固定間隔のサンプリングでは、途中への1 byte挿入だけで後続の標本位置がすべてずれる。内容依存サンプリングは、この位相ずれを避けながら大きいファイルの特徴数を抑えるために使う。

類似度は次式とする。

```text
similarity = 1 - hamming_distance(left, right) / 64
```

## 3. LSH候補抽出

閾値から許容ハミング距離`d`を求め、64 bitを`d + 1`個のバンドへ分割する。同じバンド値を1つ以上持つファイルだけを候補にし、候補へ正確なハミング距離を計算する。

距離`d`以内の2値は、`d + 1`バンドの少なくとも1つが完全一致する。この構成では、閾値内の候補をLSH段階で取りこぼさない。閾値0.0は距離64を許すため、全件比較へ切り替える。

## 4. 暫定閾値とグループ化

既定値は`61 / 64 = 0.953125`、すなわちハミング距離3以下とする。これは誤警告を抑えるための暫定値であり、呼び出し側は0.0〜1.0の範囲で上書きできる。

- BLAKE3が一致する2件以上を`exact`グループにする
- BLAKE3が異なり、SimHash類似度が閾値以上のペアを辺として連結成分を`similar`グループにする
- `similar`メンバーの`similarity`には、同じ連結成分で最も近い他ファイルとの値を保存する
- 同じ完全一致ペアだけでは`similar`グループを作らない
- 完全一致ペアが第三の類似ファイルを介して同じ`similar`連結成分へ入る場合は両方を掲載し、互いを最も近い相手として類似度1.0を保存する
- 結果順とメンバー順はファイルIDで決定的にする

SQLiteの全件再計算は読み込みから`duplicate_groups` / `duplicate_members`の置換までを1トランザクションで行う。失敗時は既存グループを保持する。

## 5. 評価計画と既知の限界

SimHashはヒューリスティックであり、正式な閾値は未決である。6科目の実データを外部送信せず、ローカル環境内だけで正解ラベル付きペアへ分類し、次を比較する。

- 完全な再ダウンロード
- ページや注記が少しだけ変わった版
- 同じ講義の別回
- 同じテンプレートを使う別科目
- 内容が異なる同名ファイル
- PDF、Office Open XML、プレーンテキスト等の形式差

閾値ごとの適合率・再現率・F1、候補数、処理時間を記録し、保存前警告と整合性画面で必要な誤警告率を決めてから既定値を確定する。

現方式は圧縮・暗号化された内容や、同じ表示内容を異なる内部表現で保存したPDF／Office文書を十分に近いと判定できない場合がある。Issue #41以降で抽出済み本文やページ単位の特徴を安全に利用できるようになった場合も、`DuplicateDetector`の入出力境界を保った別方式として比較し、実測なしに置き換えない。
