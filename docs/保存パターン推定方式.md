# 保存パターン推定方式

最終更新: 2026-07-18

この文書は、初期セットアップで既存ファイル構成から保存パターン候補を生成する方式と、方式を安全に比較・切り替えるための境界を定める。

## 目的と責務

保存パターン推定は、走査済みのファイル一覧から「今後使用する保存構成」の候補を生成し、ユーザーへ提示するために行う。既存ファイルの所属セクションを確定したり、ファイルを自動移動・削除したりする処理ではない。

新規保存時にMoodleのコース・セクション情報とユーザー確認済み設定を使って保存先を順位付けする処理は別責務とし、初期スキャン用の推定器へ混在させない。

## 共通インターフェース

すべての推定方式は`PatternEstimator`を実装し、次の同じ入出力を使用する。

```text
入力:  &[FileEntry]
出力:  EngineResult<Vec<SavePatternGuess>>
```

`FileEntry.relative_path`を推定の正本とし、OSやユーザーごとに異なる絶対パスは分析しない。方式固有の設定や学習済み状態は推定器の実装型が保持する。呼び出し側は方式固有型を扱わず、`PatternEstimator`を介して候補を取得する。

`SavePatternGuess`は次の情報を分離して保持する。

- `directory_template`: DBの保存ルールへ利用できるディレクトリ用テンプレート。`{filename}`を含めない
- `file_name_template`: ファイル名の規則を検出した場合だけ設定する比較評価用テンプレート。例は`{section}_{filename}`
- `confidence`: 方式内で候補を並べるための確からしさ
- `matched_count`: その候補を支持した既存ファイル数

今回のPRでは`file_name_template`をDBへ保存せず、推定方式の精度比較に必要な情報としてエンジン内に保持する。Tauriコマンド、候補表示UI、設定の永続化は後続issueの責務とし、外部API契約は変更しない。

## 対応するディレクトリ構成

現在は走査起点から見て次の2構成を対象とする。

| 既存構成 | `directory_template` |
|---|---|
| `科目/元ファイル名` | `{course}` |
| `科目/第4回/元ファイル名` | `{course}/第{section}回` |

セクションフォルダは認識した表記を維持し、`Week 4`なら`{course}/week {section}`のように候補化する。

走査起点直下のファイルは、走査起点自体を科目フォルダと誤認しないため候補化しない。年度・学期等を含む追加階層や任意の中間フォルダは、コース位置を安全に特定する基準が未決定のため今回の対象外とする。

## 組み込み方式

| 設定キー | 方式 | 使用する根拠 | 特徴 |
|---|---|---|---|
| `folder_only` | フォルダ限定 | 明示的な科目・セクション親フォルダ | 誤認しにくいが、ファイル名の規則を捉えない |
| `frequency` | 単純頻度 | フォルダとファイル名を同じ1票で集計 | issue #38当初実装との互換性がある |
| `evidence_weighted` | 根拠別重み付け | 根拠の明示性に応じて寄与を変更 | 数字だけのファイル名を弱い根拠として扱う |

既定方式は互換性維持のため`frequency`とする。比較評価後に既定を変更する場合も、設定キーと共通インターフェースは維持する。

### 根拠別重み付けの暫定値

| 根拠 | confidenceへの寄与 |
|---|---:|
| `第4回`等の明示的な親フォルダ | 1.0 |
| `第4回_資料.pdf`等の明示的な接頭辞 | 0.6 |
| `04_資料.pdf`等の数字だけの接頭辞 | 0.25 |

これらは比較検証用の暫定値であり、ユーザー選択結果を用いた評価後に調整する。異なる方式の`confidence`は意味が異なるため、方式をまたいで数値だけを直接比較しない。

## 候補の採用と並び順

- 走査ファイルが1件以下の場合は候補を返さない
- 走査ファイルが2件の場合は1件以上、3件以上の場合は2件以上の支持を候補条件とする
- `confidence`は全走査ファイル数を分母とする。重み付け方式では重み付き支持量を分子とする
- `matched_count`は重み付けの有無にかかわらず、実際に支持したファイル数とする
- `confidence`、`matched_count`、`directory_template`、`file_name_template`の順で決定的に並べる

閾値と重みは比較実験のためコード内の名前付き定数で管理する。変更する場合はfixtureと本書を同時に更新する。

## 切り替え方法

組み込み方式は安定キーから選択できる。

```rust
let engine = ConfigurableScanEngine::new(PatternEstimatorKind::EvidenceWeighted);
let guesses = engine.estimate_patterns(&entries)?;
```

将来のベイズ推定・オンライン学習方式は`PatternEstimator`を実装し、次のように注入する。

```rust
let engine = ConfigurableScanEngine::with_estimator(learned_estimator);
```

ランタイム設定をSQLiteへ保存する場合は`app_settings`等に`PatternEstimatorKind::key()`の値を保存する。設定値の永続化と設定UIは後続issueで実装する。

## エラーと警告

`ScanEngine::scan`は取得できたファイルと非致命的な警告を`ScanSnapshot`へまとめて返す。走査起点が存在しない、フォルダでない、または走査起点自体を読めない場合は処理全体を失敗させる。

配下の一部ファイル・フォルダだけを読めない場合は、取得できたファイルを残して走査を継続する。警告には走査起点からの相対パスと利用者向けの理由だけを保持し、環境依存の絶対パスを含めない。Tauri/UIでの警告表示方法は後続issueで決める。

## 将来方式の追加条件

新しい方式は次を満たすこと。

1. `PatternEstimator`の共通入出力を変更しない
2. 候補を確からしさ順かつ決定的に返す
3. `confidence`を0.0〜1.0に収める
4. 根拠不足時は空候補を返し、無理に決定しない
5. 方式固有の学習状態をSQLite等のローカル領域だけで扱う
6. 外部へファイル情報や学習履歴を送信しない
7. 既存ファイルの自動移動・自動削除を行わない
8. 推定根拠とユーザー確認を区別する

## 評価方針

同一の走査fixtureを各方式へ入力し、次を比較する。

- ユーザーが選択した候補が上位何位に入ったか
- 誤った第1候補を出した割合
- 候補を出せず棄却した割合
- コース別構成やフラット構成への対応率
- ファイル名規則を検出する精度
- 同じ入力に対する再現性

評価データが十分に集まるまでは方式を自動切り替えず、開発設定または明示的なユーザー設定で選択する。
