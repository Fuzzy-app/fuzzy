# 開発判断・継続時の前提

最終更新: 2026-07-20

この文書は、仕様書・Issue・実装の間で判断が必要になった際に、チームとAIが共有する合意事項を残す。機能の正規仕様は `仕様書.md`、個別タスクの完了条件はGitHub Issueを参照する。

## 判断の優先順位

1. 実装の着手・完了判定はGitHub Issueの記述を優先する。
2. Issueと `docs/仕様書.md`、API契約、DB設計の間に相違を見つけた場合、実装者は相違点を報告する。黙って仕様を上書きしない。
3. 型・API・DBスキーマを変更する場合は、従来どおり `packages/shared/src/types.ts`、`docs/api/contract.md`、`crates/engine-core/fixtures/schema.sql` の整合を確認する。

## 担当とレビュー

- subaruは主にバックエンドと全体コードレビューを担当する。
- 拡張機能とnative-hostの接点を変える際は、担当をまたぐためPRで明示して相談する。

## Moodle資料の取得と保存

- 対象は和歌山大学のMoodleである。年度によりページ構造が変わることを前提に、特定年度の固定DOMだけに依存しない解析を目指す。
- 認証が必要なMoodle資料は、ログイン済みブラウザ拡張機能が取得する。native-hostがブラウザの認証情報を保持・再利用して取得する設計にはしない。
- 取得済み資料をnative-hostへ渡して保存処理を行う。Native Messagingのサイズ制限を踏まえた実際の受け渡し形式（バイト列、一次ファイル等）は未決定であり、APIを確定するIssueで決める。
- ユーザーが保存対象と保存先を確認し、保存操作を行った後は、必要な保存先フォルダを自動作成してよい。保存済みファイルの自動移動・自動削除は引き続き禁止する。

## PR #93 / Issue #91 のスコープ判断

- Moodle再ログイン補助は単独のクリック自動化ではなく、ログイン前後の保存パネル状態、明示ログアウト後の抑止、ページ起動判定までを一つの利用者フローとして扱う。この範囲の変更はIssue #91の受け入れ対象とする。
- 同じPRに含まれるルール整合性画面、API・型・DBスキーマの整合、保存パネルの狭幅表示・開閉ハンドル改善は、PR #93でまとめてレビュー・受け入れる判断とする。以後同規模の複数機能を扱う場合は、PR本文にIssueごとの変更一覧、仕様・型・API・DBへの影響、個別のテスト結果を記載する。
- 認証補助の実装責務は、ページ判定（URL・DOM）、ログアウト抑止状態（sessionStorage・browser.storage.local）、操作調停（クリック・遷移）、UI起動判定へ分ける。一つのモジュールへDOM判定・永続化・UI起動を集約しない。
- Moodle未ログインページでは再ログイン対象と判定した画面以外の処理を起動しない。Moodle障害時のキャッシュ表示境界は `仕様書.md` 3.5を正とする。

## 外部連携

- データは原則ローカル完結とする。
- Googleカレンダー／Google Tasksへの直接追加は、ユーザーが認証し、送信対象を確認して明示的に許可した場合に限る例外とする。許可なしでは外部送信を行わず、Google認証の解除と連携停止を可能にする。
- Fuzzy独自のスマートフォン向けプッシュ通知は実装せず、スマートフォンでの通知はユーザーが許可したGoogle連携先に委ねる。
- 将来はWindows通知センターとのデスクトップ通知連携を追加する。常駐方式と、ブラウザ通知との重複防止は専用Issueで決定する。
- 発表時点では全機能の完成を目標とする。ただしGoogle連携は優先度を下げ、まずローカル機能とICS出力を完成させる。

## 未決の設計判断

- DBファイルのパス決定はIssue #36の「`app_settings` または設定ファイル」という記載と、現実装の「環境変数 `FUZZY_DB_PATH` またはOS既定データディレクトリ」に差がある。DBを開く前には `app_settings` を読めないため、設定ファイルを追加するか、Issueを現実装に合わせるかを決める。
