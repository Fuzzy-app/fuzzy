# Issue #40 作業レポート

作成日: 2026-07-22

## タスク

`crates/engine-core`の`DuplicateDetector`へBLAKE3完全一致、SimHash / LSH類似検出、SQLiteの`duplicate_groups` / `duplicate_members`再登録を実装する。

完了条件は次の3点。

- BLAKE3によるハッシュ計算と完全一致検出
- SimHash / LSHによる類似度計算（閾値は暫定）
- seedの`第4回_正規化.pdf`と再ダウンロード分を自動テストで検出

参照した正本は`docs/仕様書.md` 1.1節・1.3節・3.3節・4.1節・6章、`docs/データベース設計.md`、`docs/api/contract.md`、`schema.sql`、`seed.sql`である。

## 解決方法

### 完全一致

- ファイルを64 KiBずつ読み、BLAKE3を`b3:<hex>`形式で返す
- SimHashも同じ読み取りで計算し、ファイル全体をメモリへ載せない
- 登録済みフィンガープリントからBLAKE3一致を類似判定より先に返す

### SimHash / LSH

- 8 byteの重なり窓を内容依存で約16分の1サンプリングし、64 bit SimHashを計算する
- 類似度は`1 - hamming_distance / 64`とする
- 許容距離`d`に対して`d + 1`バンドを使い、閾値内の候補をLSHで取りこぼさない
- 暫定既定値は距離3以下（`61 / 64`）とし、定数と呼び出し引数で変更可能にした
- 完全一致グループと、閾値以上のペアによる類似連結成分を決定的な順序で生成する

詳細と実データ評価計画は[`../重複検出方式.md`](../重複検出方式.md)へ分離した。

### SQLite

- `database/duplicates.rs`へフィンガープリント読み込み、単一ファイルの再計算、グループ全件再登録を分離した
- 全件再登録は`IMMEDIATE`トランザクション内で行い、判定・制約検証・書き込みの失敗時に既存グループを残す
- `u64` SimHashはビット列を保持したままSQLiteの`i64`へ写像する
- v0→v1マイグレーションを実装し、`duplicate_members.similarity`の0.0〜1.0制約を既存DBにも適用する。範囲外の既存値があれば削除せず移行全体を中止する

## 責務と安全境界

- `duplicate.rs`は公開トレイト、保存前照合、グループ化の調停だけを担当する
- ファイル読み取り・フィンガープリント計算は`duplicate/fingerprint.rs`、LSHは`duplicate/lsh.rs`へ分割し、`duplicate.rs`には公開境界とグループ化の調停を残した
- `database/duplicates.rs`はSQLiteの読み書きとトランザクションだけを担当する
- Native Messagingのwire型、APIコマンド、DBテーブル形状、共有TS型は変更していない
- 検出は提示・警告用であり、ファイルの移動・削除・書き換えを行わない
- 閾値、重複ID、空ハッシュ、非有限・範囲外の類似度を境界で拒否する

## テスト

- 同一内容の別名ファイルがBLAKE3完全一致になる
- 少量変更した内容のSimHashは暫定閾値以上、無関係な内容は閾値未満になる
- 保存前照合が完全一致と近傍一致だけを類似度順に返す
- LSHが暫定閾値境界（距離3）のペアを候補から落とさない
- 完全一致ペアを類似グループへ二重登録しない
- NaN等の不正閾値と重複ファイルIDを拒否する
- 空ファイルのフィンガープリントが決定的になる
- seedのfile id 3 / 9を完全一致グループとして再登録する
- seedの仮SimHashを64 bit全体へ分散させ、完全一致ペア以外を誤って類似グループにしない
- 判定失敗時に既存DBグループを保持する
- フィンガープリント更新でファイル内容を変更しない
- v0→v1移行が類似度制約を追加し、不正な既存値ではロールバックする

## 残る接続境界

`checkSimilarFiles`のNative Messagingコマンドは現時点で未実装であり、wire payloadはMoodleの`fileMeta`だけを持つ。認証済みブラウザが取得した資料をどの時点でローカル一時ファイルとして`DuplicateDetector`へ渡すかは、native-hostコマンド実装IssueでAPI接点として合意する必要がある。本Issueでは既存APIを独断で変更せず、ローカルファイルを受け取った後の共有エンジンとSQLite登録までを完了した。
